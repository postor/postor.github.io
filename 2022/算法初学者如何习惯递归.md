# 如何熟悉递归

老婆在练习算法，问我怎么就是学不好递归呢？这个问题我也蒙住了，递归不就是递归么，会用递归算斐波那契就是会递归了吧，剩下的是熟练度么？熟练度也差不多就行吧，测试/侧开的话不需要算法玩的很溜吧，我是这样想的，测试侧开要胜任本职工作应该更重视全面性，比如各平台、各类型的测试，各种工具/接口的熟练度，升华职能应该是辅助项目管理优化工作流程，测试早介入多规范会有很多好处的；毕竟纯项目管理大多是单纯的敏捷，看到的进度是可工作的软件，测试对项目的把控比单纯的看着能跑更加深入。

话说回来对测试/侧开来说“差不多的程度”还是必要的，但凡写代码功能测也好、工具脚本也好，经常还是有虽然循环能做到但递归更快更整洁，或者偶尔不递归解决不了的情况呢

对于已经熟悉递归的同学使用递归就是理所当然的事情呢，感觉像数学定理一样已经不用解释了，但刚接触的时候它确实是个门槛，我就试着用自己觉得更好理解的方式分析一下递归吧

## 递归和循环的关系

先讲一下这个有点无聊的话题，如果确实觉得这个无聊可以直接跳到下一小节。

- 一个题目如果循环可以解决，那么递归同样可以解决
- 任意的递归问题，同样可以转化为循环来解决
- 多写一层循环代表多一个维度，递归则可以一次解决无限个维度

1.一个题目如果循环可以解决，那么递归同样可以解决

举个简单的循环例子

```
for(let i=0;i<100;i++){
  console.log(i)
}
```

转换成递归实现的话

```
function loop(x){
  if(x<100){
    console.log(x)
    loop(x+1)
  }
}
loop(1)
```

需要注意的是每次递归调用会增加堆栈深度，比如上面例子如果循环次数非常大，递归就回报堆栈超限错误

2.任意的递归问题，同样可以转化为循环来解决

例如遍历二叉树

```
function travel(node){
  if(!node) return
  console.log(node.value)
  travel(node.left)
  travel(node.right)
}
```

换成循环的话

```
function travel(node){  
  let queue=[node]
  while(queue.length){
    let n = queue.shift()
    console.log(node.value)
    n.right && queue.unshift(n.right)
    n.left && queue.unshift(n.left)
  }
}
```
转换意味着空间的腾挪会比直接递归浪费时间空间

3.多写一层循环代表多一个维度，递归则可以一次解决无限个维度

比如用循环遍历 3 维数组，可以假设应用场景是我们在找三维空间里的物品或者确认魔方每个颗粒的状态

```
function travel(arr){
  for(let x=0;x<arr.length;x++){
    for(let y=0;y<arr[x].length;y++){
      for(let z=0;z<arr[x][y].length;z++){
        ....
      }
    }
  }
}
```

换成递归

```
function travel(arr,path=[]){
  if(!Array.isArray(arr)){
    let [x,y,z] = path
    ....
    return
  }
  for(let i=0;i<arr.length;i++){
    travel(arr[i],path.concat([i]))
  }
}
```
可以看出不论是四维五维或者多少递归都是不用改什么代码的

## 递归的核心思路

递归的核心思路是把大的问题不断拆分成小的问题，直到问题简单到很直白的程度

示例1：树/图遍历

树的任意节点左子树和右子树也都是树，所以遍历整个树就可以拆分成【节点本身】+【左子树】+【右子树】

图和树的不同并不很大，可以简单理解成多个子节点的树外加可能有环，【多个子节点】和【左右子节点】差距并不大，【环】的问题则可以通过全局标记访问过的节点来避免重复

示例2：背包问题

https://baike.baidu.com/item/%E8%83%8C%E5%8C%85%E9%97%AE%E9%A2%98/2416931

背包问题理所当然的解法是动态规划，但就我理解，动态规划对初学者不太友好，而且动态规划也可以理解为是从递归进一步抽象总结的结果，就先从思路说吧，第一层

- 往背包里一个一个的放，不断尝试，直到找到最优解
- 先全放到背包里，一个一个往外拿，不断尝试，直到找到最优解

其实两种没差多少，就选第一个方案吧，如何拆分成小问题呢，其实很简单，10个东西随便装进一个之后就变成9个了，然后是8个7个....直到装不下，我们就知道所有情况

```
/**
 * limit 背包容量上限
 * stones 宝石列表，每个宝石有[大小,价值]两个属性
 */
function pack(limit=100,stones=[[5,100],[77,92],...]){
  let max=0,usedStones=[]
  // 递归开始于一个宝石也没有使用
  pack(stones.map(x=>false),0,0)
  return [max,usedStones]
  /**
   * 每次选择一个装进包里，递归
   * used 已经使用石头的标记
   * size 当前占用的容量
   * value 当前背包价值
   */
  function pack(used=[],size=0,value=0){
    for(let i=0;i<stones.length;i++){
      if(!used[i]){
        let s = size+stones[i][0]
        if(s>limit) return // 超限
        let v = value+stones[i][1]
        let u=[...used]
        u[i]=true
        // 如果找到更大的组合则更新结果
        if(v>max){
          max=v
          usedStones=u
        }
        // 继续找下一个石头
        pack(u,s,v)
      }
    }
  }
}

```

递归效率不如动态规划的地方主要在于重复的运算，比如同样的几块石头，动态规划只计算了一次，而递归就回重复算许多次

## 总结

递归总的来说应该能在中高难度的算法题里 70% 的场景会用到，剩下的是动态规划、位运算和一些奇思妙想，所以递归不是那么容易上手也是正常的，毕竟递归熟练了基本中等难度就算解决，如果本文帮不上忙的话就多练题吧，有的人就是要熟练度先上来再去理解呢。
